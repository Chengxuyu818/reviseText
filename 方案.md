# 英语作文自动修改网站详细方案书（大学生个人项目版）

## 1. 项目定位

### 1.1 项目名称（暂定）
**EssayFixer**（英语作文自动修改网站）

### 1.2 项目目标
你作为大学生开发者，做一个可上线、可演示、可持续迭代的网站：
- 用户粘贴英语作文后，系统自动识别错误并高亮修改建议；
- 严格按三色规则展示：
  - **拼写错误：红色**
  - **用词错误：蓝色**
  - **语法错误：绿色**
- 输出“原文 + 修改建议 + 修订版 + 错误统计”，形成学习闭环。

### 1.3 项目价值
- 对用户：快速改作文、看得懂、学得会；
- 对你本人：可作为课程设计、毕业设计、求职作品集（含前端、后端、AI调用、可视化）。

---

## 2. 需求定义

### 2.1 核心需求（必须实现）
1. 用户输入英语作文（建议 80~800 词）。
2. 系统识别错误并分类：拼写 / 用词 / 语法。
3. 页面中按三色高亮错误，并显示建议替换词句。
4. 提供修订后全文。
5. 提供错误数量统计（各类型数量、总错误数、错误率）。

### 2.2 规则定义（关键）
- **红色（Spelling）**：单词拼写错误，如 `enviroment -> environment`。
- **蓝色（Word Choice）**：搭配不当、词义不准、表达不自然，如 `do a decision -> make a decision`。
- **绿色（Grammar）**：时态、主谓一致、冠词、介词、从句结构等语法错误。

> 注：同一片段如同时命中多个错误，优先级建议为：**语法 > 用词 > 拼写**，避免颜色冲突。

### 2.3 非功能需求
- 响应时间：单篇 500 词内尽量控制在 3~8 秒；
- 可用性：界面简单，移动端可浏览；
- 可扩展：后续可加账号系统与历史记录。

---

## 3. 用户流程设计

1. 用户进入首页，点击“开始批改”。
2. 在输入框粘贴作文，点击“分析并修改”。
3. 系统调用批改服务，返回结构化错误列表。
4. 前端渲染：
   - 原文中错误片段按红/蓝/绿高亮；
   - 侧栏展示每条错误解释与建议；
   - 底部展示修订版全文。
5. 用户复制修订版，或继续编辑再提交。

## 4. 功能模块设计

## 4.1 MVP 功能（第一阶段）

### 模块 A：作文输入
- 大文本输入框（支持粘贴）
- 字数统计、最小字数提示
- 提交按钮与加载状态

### 模块 B：三色批改展示（核心）
- 原文高亮标注：
  - 拼写错误：红色
  - 用词错误：蓝色
  - 语法错误：绿色
- 鼠标悬停（或点击）错误片段显示：
  - 错误类型
  - 错误原因
  - 建议改法

### 模块 C：修订版输出
- 自动生成修订版全文
- 支持一键复制

### 模块 D：统计面板
- 本文错误总数
- 按类别统计数量
- 简单建议（如“语法错误偏多，重点复习时态和从句”）

## 5. 技术方案

## 5.1 技术选型（适合大学生）

### 前端
- **Next.js + TypeScript + Tailwind CSS**
- 富文本/高亮渲染：可用 `react-markdown` + 自定义 span 标注，或直接用 HTML 片段渲染

### 后端
- **Node.js + Express**（简单上手）
- 或 **NestJS**（结构更规范）

### AI 与规则
- 大模型 API（如 OpenAI 兼容接口）负责语义与语法分析
- 拼写可叠加词典校验（如 `nspell` / `typo-js`）提高稳定性

### 数据存储
- MVP 可先不接数据库（仅在线体验）
- 进阶可加 PostgreSQL 存历史记录

### 部署
- 前端：Vercel
- 后端：Render / Railway / 云服务器
- 域名+HTTPS：可后期接入

---

## 5.2 系统架构
- 前端页面：输入、结果渲染、统计展示
- 后端 API：接收文本、调用批改引擎、返回 JSON
- 批改引擎：规则层 + LLM层融合

数据流：
1. 前端提交作文文本
2. 后端预处理（分句、清洗）
3. 调用批改逻辑得到错误列表
4. 返回结构化数据给前端
5. 前端按错误区间上色渲染

## 6. 三色规则落地实现（重点）

## 6.1 返回数据结构（建议）
后端统一返回：
- `originalText`
- `revisedText`
- `errors[]`：每条包含
  - `start`
  - `end`
  - `type`（spelling | word_choice | grammar）
  - `wrong`
  - `suggestion`
  - `reason`

## 6.2 颜色映射规则
- `spelling -> #ef4444`（红）
- `word_choice -> #3b82f6`（蓝）
- `grammar -> #22c55e`（绿）

## 6.3 冲突处理
若区间重叠：
1. 按优先级保留高优先级（grammar > word_choice > spelling）；
2. 或将重叠项拆分区间后分别渲染；
3. 在侧栏保留全部错误说明，避免信息丢失。

## 6.4 前端渲染策略
- 先将文本切分为 token/片段；
- 将错误区间映射到片段；
- 用 `<span class="text-red-500">` 等方式高亮；
- hover 时显示 tooltip（错误原因 + 建议）。

## 7. API 设计（MVP）

### 7.1 批改接口
- `POST /api/correct`

请求：
- `text`: string
- `level`: string（可选，如 college / ielts）

响应：
- `originalText`
- `revisedText`
- `stats`
  - `total`
  - `spellingCount`
  - `wordChoiceCount`
  - `grammarCount`
- `errors[]`

### 7.2 健康检查
- `GET /api/health`

用于部署后监控服务是否正常。

## 11. 风险与应对

1. **模型结果不稳定**
   - 应对：固定返回JSON格式，失败时重试或降级。

2. **颜色冲突和误标**
   - 应对：统一优先级策略 + 人工样例回归测试。

3. **调用成本上升**
   - 应对：限制单次字数、增加免费额度上限、缓存重复请求。

4. **上线后访问慢**
   - 应对：后端加队列、限制并发、优化接口超时策略。
